你大概率是在说 **RankMixer**（你写的 rankingmixer 多半是顺手写法），里面的 **token mixing** 模块。它不是做文本生成的模型，而是面向**工业推荐 / 广告排序（ranking）**的模型架构：论文把推荐系统里的大量异构特征做成一组 **feature tokens**，再用 RankMixer block 做特征交互，目标是在延迟基本不变的前提下，把排序模型做得更大、更适合 GPU。

  

这里的 **token mixing**，不是 self-attention 那套 QK^\top 相似度计算。RankMixer 先把输入特征 tokenization 成 T 个 feature tokens；然后在一个 **Multi-head Token Mixing** 里，把每个 token 的 embedding 均分成 H 个 heads，再把“所有 token 的第 h 个 head”拼接成一个新的 mixed token。论文写法是

s^h = \mathrm{Concat}(x_1^h, x_2^h, \dots, x_T^h).

直觉上，就是把不同特征 token 的“同一子空间切片”重新洗牌、重组，让用户特征、物品特征、序列特征之类发生交互。

  

所以可以把它粗略理解成：**“给推荐排序模型做的一种更便宜的 token 级特征交互器”**。作者明确说，他们用 multi-head token mixing 来替代二次复杂度的 self-attention；原因是推荐系统里的特征空间天然异构，直接拿不同语义空间做 attention 内积并不理想，而且 self-attention 更耗算力、内存 IO 和显存。一个 **RankMixer block** 就是 **Token Mixing + Per-token FFN**，外加残差和 LayerNorm。

  

一句人话版：

  

**RankMixer = 先把推荐系统特征变成 token，再用“切分—重排—拼接”的 token mixing 来做特征交互的排序模型。** 后续 2026 年的 **TokenMixer-Large** 论文还明确说，前一版的 TokenMixer 架构就是在 RankMixer 论文里引入的，只是后来又继续往更深、更大参数量扩展了。
